---

PRの差分を分析し、実装上の仕様変更を「追加・変更・削除」に分類して一覧化します。

**重要な前提**: このスキルが出力するのは「コードから読み取れる実装上の仕様」です。これが「真に満たすべき仕様」かどうかの判断は人間が行います。実装を正として扱わず、あくまで実装されている内容の要約として読んでください。

## 引数

- `$ARGUMENTS`: PR番号またはPR URL（必須）

## 手順

### 1. PR情報の取得

1. `gh pr view $ARGUMENTS` でPRのタイトル・説明を取得する
2. `gh pr diff $ARGUMENTS` でPRの差分を取得する
3. 差分が大きすぎる場合は `gh pr diff $ARGUMENTS --name-only` で変更ファイル一覧を取得し、ファイル単位で段階的に分析する

### 2. テストコードの分析（最優先）

振る舞いの調査は**テストコードから着手**する。テストはプロダクションコードより仕様の意図が明確に表現されていることが多い。

1. 差分に含まれるテストファイルを先に読む
2. テストケース名・テストデータ・アサーションから仕様を読み取る
3. 以下の注意点を常に意識する：

**テストに関する注意点**:
- **テストが実装に合わせて無理やり通されている場合がある**: テストが仕様を表現しているとは限らない。アサーションの値が不自然でないか、テストケース名と実際のテスト内容が乖離していないかに注意する。違和感がある場合は `[テスト要確認]` を付ける
- **テストが実装されていないロジックがある**: テストがあるコードだけでなく、テストがカバーしていない実装コードの変更も漏れなく分析する。テストのないロジックには `[テスト未実装]` を付ける

### 3. 差分の分析

テストコードの分析結果を踏まえて、実装コードの差分を読む。以下の観点で仕様レベルの変更を抽出する：

- **振る舞いの変更**: 入出力・条件分岐・エラーハンドリング・バリデーションの追加や変更
- **データモデルの変更**: フィールドの追加・削除・型変更・制約変更
- **APIの変更**: エンドポイント・パラメータ・レスポンス形式の変更
- **設定・構成の変更**: 設定値・デフォルト値・依存関係の変更
- **UIの変更**: 画面要素・操作フロー・表示内容の変更

以下は仕様として抽出しない：
- 純粋なリファクタリング（外部から見た振る舞いが変わらないコード変更）
- コメントやドキュメントの修正
- テストコード単体の変更（ただしテストから読み取れる仕様は上のステップで参考にする）
- import文の整理

### 4. 出力

分析結果をMarkdownファイルとして出力する。

1. リポジトリルートに `pr-spec-<PR番号>.md` というファイル名で書き出す
2. 該当がないカテゴリは省略する
3. ファイルパスをユーザーに表示する

ファイルの内容は以下のフォーマットに従う：

```markdown
# PR仕様サマリー

**PR**: #番号 タイトル
**説明**: PRの説明から要約（1〜2文）

## 追加された仕様
- 仕様の説明（対象: ファイルや関数など影響範囲）
- ...

## 変更された仕様
- 変更前 → 変更後の説明（対象: 影響範囲）
- ...

## 削除された仕様
- 削除された仕様の説明（対象: 影響範囲）
- ...
```

## 出力ルール

- 各仕様は1行で簡潔に記述する
- 技術的な実装詳細ではなく、振る舞いや機能としての仕様を記述する
  - 悪い例: `UserService.validate()メソッドにnullチェックを追加`
  - 良い例: `ユーザー登録時に名前が空の場合はバリデーションエラーを返す`
- 影響範囲を括弧で付記し、レビュアーがコードを特定できるようにする
- 不明確な変更には末尾に `[要確認]` を付ける
- 出力ファイルは一時的なレビュー用であり、リポジトリにコミットしない
