Gitコンフリクトの原因を特定し、どのように解決するかをユーザーに確認してから解消する。自己判断での変更は行わない。

## 手順

### 1. コンフリクトの検出

`git diff --name-only --diff-filter=U` でコンフリクトが発生しているファイルを一覧化する。

### 2. 原因の分析

コンフリクトファイルごとに以下を分析する:

- コンフリクト箇所の内容（`<<<<<<<` / `=======` / `>>>>>>>` マーカー）
- **ours（現在の変更）**: 何をどう変更したか
- **theirs（リモートの変更）**: 何をどう変更したか
- コンフリクトの性質（同一行の編集 / 近接行の編集 / ファイル削除と編集の競合 等）

### 3. 解決方針の提案

ファイルごとに以下の選択肢を提示する:

- **ours を採用**: 現在の変更を優先する
- **theirs を採用**: リモートの変更を優先する
- **手動マージ**: 両方の変更を統合する（具体的なマージ結果の案を提示する）

複数ファイルがある場合はまとめて提示し、1回の確認で済ませる。

### 4. ユーザーの承認

**必ずユーザーの明示的な承認を得てから**解決を実行する。

### 5. コンフリクト解消の実行

承認された方針に従い、各ファイルのコンフリクトを解消する。

### 6. 確認

- `git diff` で解消結果を表示する
- 残りのコンフリクトがないことを `git diff --check` で確認する

## 注意

- **自己判断での変更は絶対に行わない**。必ずユーザーに確認する
- コンフリクト箇所が多い場合はファイル単位でグルーピングして提示する
- マージコミットの作成やステージングは、ユーザーが指示するまで行わない
