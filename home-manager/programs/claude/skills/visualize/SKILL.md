ドキュメントや会話内容を分析し、**読者が「現在地」と「全体像」を掴めること**を最優先に図を生成します。

**設計思想**:
- 複雑なものを複雑なまま図にしない。本質的な3〜5の概念を抽出して単純化する
- 全体像 → 現在地 → 詳細 の順で説明する（ズームイン方式）
- 「自分が何を知らないかわからない」読者のために、まず地図を渡す
- 認知負荷の最小化: 1図あたり最大7要素（Miller's Law: 人間のワーキングメモリは7±2チャンク）

## 引数

- `$ARGUMENTS`: 可視化対象の指定（以下のいずれか）
  - ファイルパス（ドキュメント、ソースコード）
  - PR番号 / PR URL
  - 「直前の出力」「上の内容」等の会話参照
  - 自由テキスト（分析対象の内容そのもの）

## 手順

### 1. 対象コンテンツの取得

引数の種類に応じて内容を取得する：

- **ファイルパス**: ファイルを読み取る
- **PR番号 / URL**: `gh pr view` と `gh pr diff` で情報を取得する
- **会話参照**: 会話履歴から該当する内容を取得する
- **自由テキスト**: そのまま分析対象とする

### 2. 読者の「わからなさ」の特定

**ここが最も重要なステップ。**「この図を見て何が理解できるか」ではなく「読者が何を理解できていないか」から出発する。

以下を特定する：

- **この対象が属する上位概念は何か**: これは何の一部か？何の仕組みの上で動くか？
- **読者がつまずく可能性が最も高い概念**は何か
- **「これさえわかればあとはわかる」核心概念**は何か（必ず1〜3個に絞る）
- **全体の中でこの対象はどこにあるか**（地図上の「現在地」）

### 3. 図の構成計画（ズームイン方式）

**必ず全体像から始めて詳細へ進む。** 以下の3層で図を計画する：

#### 層1: 全体マップ（必須）
「この対象が宇宙のどこにあるか」を示す地図。
- この対象を取り囲む上位概念・外部世界を描く
- 対象自体は1つのブロックとして扱い、内部は描かない
- 「今回説明するのはここの話です」という現在地マーカーを明示する
- 読者が既に知っているはずの概念（既知の錨）を必ず1つ以上含める

例: Nixを説明するなら「パッケージマネージャーという概念全体 → apt/brew/Nixの位置づけ → Nixの中のFlake」という地図を先に見せる

#### 層2: 構造図（対象によって選択）
対象そのものの構造・関係性を、要素を絞って示す。
- **最重要概念のみ**: 全要素を網羅しない。「これさえわかれば理解が進む」概念を3〜5個選ぶ
- 省略した要素は図の注釈に「他にもN個あるが省略」と書く

#### 層3: 詳細図（必要な場合のみ）
特定の処理フローやインタラクションを詳細化する。
- 層2の理解があることを前提に書く
- 「層2のどの部分を掘り下げているか」を図のタイトルで明示する

### 4. 簡略化の原則

**複雑な対象ほど、図はシンプルにする。**

- 要素数が7を超えたら必ず分割・グループ化を検討する
- 「全部書かないと正確じゃない」という誘惑に負けない。正確さより理解を優先する
- 「AはBに依存する」という関係が10本あるなら、最も重要な3本だけ書いて残りは省略する
- 例外・エッジケース・詳細仕様は図に入れない（補足テキストへ）

### 5. 可視化手法の選択

分析結果に基づき、各層に最適な手法を選ぶ。

#### 全体マップ（層1）に適した図

| 図 | 構文 | いつ使うか |
|----|------|-----------|
| **C4コンテキスト図** | `C4Context` | システム・アーキテクチャの全体像 |
| **マインドマップ** | `mindmap` | 概念の位置づけ・分類体系 |
| **フローチャート（概念図）** | `flowchart TD/LR` | 「A の中に B があり、今は B の話」という包含関係 |

#### 構造図（層2）に適した図

| 図 | 構文 | 伝える情報 |
|----|------|-----------|
| **フローチャート** | `flowchart TD/LR` | 処理フロー、依存関係、コンポーネント構成 |
| **クラス図** | `classDiagram` | 型の関係性（継承・集約） |
| **ER図** | `erDiagram` | データモデルのリレーション |
| **ブロック図** | `block-beta` | レイヤー構成、モジュール間の構造 |
| **C4コンテナ図** | `C4Container` | システム内の技術的構成要素 |

#### 詳細図（層3）に適した図

| 図 | 構文 | 伝える情報 |
|----|------|-----------|
| **シーケンス図** | `sequenceDiagram` | 時系列のメッセージフロー |
| **状態遷移図** | `stateDiagram-v2` | ライフサイクル・状態変化 |
| **アクティビティ図代替** | `flowchart` | 処理の詳細な分岐・並列 |

#### 選択の原則

1. **全体から詳細へ**: 層1（地図）を省略しない
2. **7要素以下**: 超えたら分割する。網羅より理解を優先
3. **既知の錨を含める**: 読者が知っているであろう概念を起点にする
4. **1図1関心事**: 複数の関心事を1つの図に詰め込まない
5. **Mermaid優先**: レンダリング可能な環境が広い

### 6. ダイアグラムの生成

#### 生成ルール

- **ノード・ラベルは日本語**: 対象ドキュメントが日本語の場合、図中のラベルも日本語にする
- **現在地マーカー**: 層1の全体マップでは、今回の対象ノードを `style X fill:#ff9,stroke:#f90` 等で強調する
- **色・スタイルの活用**: 重要な箇所、変更箇所、問題箇所をスタイルで強調する
- **凡例の付与**: 色やスタイルで意味を持たせた場合は凡例を付ける
- **方向の選択**: 処理フローは TB（上→下）、時系列・データフローは LR（左→右）を基本とする

#### Mermaid の記法上の注意

- ノードIDに日本語を使わず、英数字のIDに日本語ラベルを付ける: `A[ユーザー認証]`
- 特殊文字（括弧、引用符等）はラベル内で使わないか、`"` で囲む
- subgraph のラベルも英数字IDと分離する

### 7. 出力

分析結果を Markdown ファイルとして出力する。

1. リポジトリルートに `visualize-<対象名>.md` というファイル名で書き出す
   - ファイルパスの場合: `visualize-scrape-timeline.md`（拡張子とディレクトリを除いたファイル名）
   - PR番号の場合: `visualize-pr-123.md`
   - 会話参照・自由テキストの場合: 内容から適切な短い名前を付ける
2. ファイルパスをユーザーに表示する

ファイルの内容は以下のフォーマットに従う：

```markdown
# [対象名] の概念マップ

> [読者へのガイド: 「これを読む前に知っておくべきこと」と「この図で何がわかるか」を2〜3行で]

## 全体像 — これはどこの話か

[1行: この対象が属する上位概念と現在地の説明]

\`\`\`mermaid
[層1: 全体マップ（現在地を強調表示）]
\`\`\`

## [構造/仕組み] — 核心となる概念

[1行: この図が何を表しているか。「ここでは〇〇の部分だけを掘り下げます」という説明]

\`\`\`mermaid
[層2: 構造図（3〜7要素に絞る）]
\`\`\`

## [詳細フロー] — （必要な場合のみ）

[1行: 層2のどの部分を掘り下げているかの説明]

\`\`\`mermaid
[層3: 詳細図]
\`\`\`

## 補足

<!-- 省略した要素、図の読み方のヒント、理解をさらに深めるための次のステップ -->
```

## 出力ルール

- テキストによる要約は最小限にし、図で語る
- 全内容を網羅しようとしない。読者が最も「わからない」部分に集中する
- 複雑な対象こそ、図はシンプルに。7要素を超えたら分割する
- 出力ファイルは一時的な可視化用であり、リポジトリにコミットしない
- ダイアグラムが正しくレンダリングされるか、Mermaid の構文を慎重に確認する

## 例

### 概念理解の可視化
```
User: /visualize Nixのflakeの仕組み
→ 層1: パッケージ管理の全体地図（apt/brew/Nixの位置づけ、今はflakeの話）
  層2: flakeの構造（inputs/outputs/lockfileの3要素に絞る）
  層3: ビルド評価フロー（必要なら）
```

### PR差分の可視化
```
User: /visualize #123
→ 層1: 変更されたモジュールがシステム全体のどこにあるか
  層2: 変更の差分構造（Before/After）
```

### 設計ドキュメントの可視化
```
User: /visualize docs/architecture.md
→ 層1: このシステムが外部世界とどう繋がるか（C4コンテキスト）
  層2: 内部の主要コンポーネント（C4コンテナ、5個以内）
  層3: 重要なフロー1本だけ（シーケンス図）
```
